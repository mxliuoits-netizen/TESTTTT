# RA006 SAP員工主檔調整需求 — 修改規劃書

> 本文件為**規劃書**，僅說明依規格書可以完成的修改方向，**目前 `UpdateMasterData` 專案程式碼尚未異動**。

## Context

SAP 端調整了提供給後勤考勤系統的員工主檔規格（需求書：`超商後勤考勤_RA006_功能規格書_SAP員工主檔調整需求_20260529`，v1.10 2026/5/29，逐節重點已整理於 `KCH\RA006_需求彙整.md`）。核心變化：

1. **員工每日在職主檔（全件）**：新增分類 `0`（數位大樓），既有 `1`商場 `2`聯服 `3`地區 `4`門市 `9`其他不變。
2. **員工異動主檔（差異，實際檔名 `rs_EMP_DIFF`）**：精簡5欄位格式 `姓名|員工編號(8碼)|異動類型(2碼)|異動生效日(8碼)|附加值`，異動類型：`01`PT→FT `02`FT→PT `06`調出超商 `09`離職 `10`留停 `11`分類異動（附加值＝新分類代碼0-9）。

**比對規則（本次改版核心）**：系統先用「在職主檔」跟既有 `Base_Employee_m` 比對出差異，**只有比對出差異時**才去「異動主檔」找對應[異動類型]的[異動生效日]，當作到職/離職/留停/調出/PT-FT轉換/分類異動的正式生效日；若在職主檔沒有差異，忽略異動主檔；若異動主檔查無對應資料，需寄警示信通知。

**已確認的關鍵程式碼事實**（`UpdateMasterData\Service\EmployeeViceService.cs`）：
- 第100-111行原本有一段開發者留下但邏輯空白的 stub（`//取出 diff 資料 20260629`），是規格書要求接續完成的核心位置。
- 現有 `JobEmployeeDiffModel`/`Job_EmployeeDiff_tmp`（`DiffStatus` I/U/D + `SapTransferCode` B8/C1/B5）經比對SAP介面規格書xlsx確認，對應到規格書明確標註「不需此檔」的舊版棄用格式；SAP實際提供的是`rs_EMP_DIFF` 5欄位新格式。
- 到職/復職/調入日「系統日 vs 生效日」比較邏輯（第113-125行），已用需求書「馬映佳」範例驗證，**現有邏輯已正確符合規格，不需修改**。

## 依規格書可以完成的修改項目

### 1. `Job_EmployeeDiff_tmp` 資料表 + Model + Dao — 支援新5欄位格式（新舊並存）

- **DB migration**（需另外執行，非程式碼本身）：在既有 `sethq.job_employeediff_tmp` 新增可為NULL的欄位 `change_type`(varchar2)、`effective_date`(date)、`extra_value`(varchar)，舊欄位保留不動（已確認實際schema無撞名疑慮）。
- **`job_config_setting_dev1` 設定變更（已確認現況，見上方「已確認的實際資料庫現況」）**：目前`Job_UpdateEmployeeDiff`設定的`FTPFileName`/`ManualFileName`是`EMP_DIFF{0}.TXT`、Col1~Col16對應舊16欄位格式。需要新增一組對應新格式的設定（建議做法：`FTPFileName`/`ManualFileName`改為`rs_EMP_DIFF{0}.TXT`，`HeaderCount`需依新表頭長度重新計算，欄位改為5欄：`Col1.Name=empname,Len=20`、`Col2.Name=empid,Len=8`、`Col3.Name=change_type,Len=2`、`Col4.Name=effective_date,Len=8,Type=date,Rule=yyyyMMdd`、`Col5.Name=extra_value,Len=1`）。**這是上線切換的關鍵一步，切換時機需與SAP端實際何時停發舊檔/改發新檔同步，否則會抓錯檔案格式導致解析失敗**。
- **`Model/EmployeeDiff/JobEmployeeDiffModel.cs`**：新增 `ChangeType`/`EffectiveDate`/`ExtraValue` 三屬性，舊屬性保留。
- **`Dao/JobEmployeeDiffDao.cs`**：新增 `GetJobEmployeeDiffMap()`，以 `(Empid, ChangeType)` 複合鍵查詢，用 `ROW_NUMBER() OVER (PARTITION BY empid, change_type ORDER BY id DESC)` 取每組最新一筆（符合規格書「後蓋前」規則），不動既有舊方法。

### 2. `EmployeeViceService.UpdateEmployee()` 核心比對邏輯 — 完成第100-111行的空白邏輯

在既有 `editEmpList`/`addEmpList`/`deleteEmpList` 三段 LINQ 架構中插入情境判斷：

- **PT/FT轉換**（`editEmpList`內，目前技術債用途的`Birthday`欄位）：`tmp_Emp.Ptft != EV.Ptft`時，查 `diffMap[(empid,"01")]`或`"02"`（依轉換方向）的`EffectiveDate`；查無資料則fallback用`JobDate`並記警示。
- **分類異動**（`editEmpList`內，`GroupType`有差異時）：查`diffMap[(empid,"11")]`的`EffectiveDate`作為身分轉換日（建議沿用`TransferUpdateTime`欄位存放，因目前模型沒有專屬欄位——**這是一個假設，需與需求方確認是否有更合適的欄位**）；`ExtraValue`（新分類代碼）與主檔`GroupType`交叉驗證，不一致時記警示。
- **主檔不存在→離職/留停/調出**（`deleteEmpList`）：依序查`diffMap[(empid,"09")]`（離職）→`"06"`（調出）→`"10"`（留停），取第一個命中當作`Expiredate`/`TransferUpdateTime`；三者皆查無資料時維持既有`jobEmployeeviceTDao.GetTransferUpdateTime(...)`的fallback（不中斷流程，只記警示）。優先順序`09>06>10`為假設，需求方未明確定義。
- **到職/復職/調入日**：不修改，已驗證符合規格。

### 3. 警示記錄機制

新增`MailTemplateType.異動主檔缺漏警示 = 301`，沿用既有`mail_normal_m` + `MailTemplateService.BuildMailContent<T>()`機制（不新建資料表）：在`UpdateEmployee()`內收集「有差異但異動主檔查無對應[異動類型]資料」或「附加值與主檔不一致」的員編/情境，流程尾端組信寫入`mail_normal_m`。`mail_template_m`需新增對應樣板文字（含規格書要求的「未有相關日期」字樣）。收件人建議透過新設定項（如`UpdateEmployee.DiffWarningMailTo`）控制，而非寫死。

### 4. 分類0（數位大樓）設定檢查清單

- **已確認**：`system_config_m`的`UpdateEmployee.TeamTypeWhiteList`、`UpdateOrganization.TeamTypeWhiteList`、`UpdateGPS.TeamTypeWhiteList` 目前皆為`1,2,3,9`，**三處都需要新增`0`**，否則數位大樓員工在白名單過濾階段就會被三個Job（員工/組織/GPS主檔）分別排除轉入。這是三筆資料異動（非程式碼），建議列入上線前部署檢查清單。
- `EmployeeViceService.cs`「後勤成本中心轉換」判斷式`(GroupType == "3" || GroupType == "9")`是否要納入`"0"`，待需求方確認數位大樓是否算後勤類，暫不擅自變更。
