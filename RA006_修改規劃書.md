## Context

SAP 端調整了提供給後勤考勤系統的員工主檔規格（需求書：`超商後勤考勤_RA006_功能規格書_SAP員工主檔調整需求_20260529`，v1.10 2026/5/29，逐節重點已整理於 `KCH\RA006_需求彙整.md`）。核心變化：

1. **員工每日在職主檔（全件）**：新增分類 `0`（數位大樓），既有 `1`商場 `2`聯服 `3`地區 `4`門市 `9`其他不變。
2. **員工異動主檔（差異，實際檔名 `rs_EMP_DIFF`）**：精簡5欄位格式 `姓名|員工編號(8碼)|異動類型(2碼)|異動生效日(8碼)|附加值`，異動類型：`01`PT→FT `02`FT→PT `06`調出超商 `09`離職 `10`留停 `11`分類異動（附加值＝新分類代碼0-9）。

**比對規則（本次改版核心）**：系統先用「在職主檔」跟既有 `Base_Employee_m` 比對出差異，**只有比對出差異時**才去「異動主檔」找對應[異動類型]的[異動生效日]，當作到職/離職/留停/調出/PT-FT轉換/分類異動的正式生效日；若在職主檔沒有差異，忽略異動主檔；若異動主檔查無對應資料，需寄警示信通知。

**已確認的關鍵程式碼事實**（`UpdateMasterData\Service\EmployeeViceService.cs`）：
- 第100-111行原本有一段開發者留下但邏輯空白的 stub（`//取出 diff 資料 20260629`），是規格書要求接續完成的核心位置。
- 現有 `JobEmployeeDiffModel`/`Job_EmployeeDiff_tmp`（`DiffStatus` I/U/D + `SapTransferCode` B8/C1/B5）經比對SAP介面規格書xlsx確認，對應到規格書明確標註「不需此檔」的舊版棄用格式；SAP實際提供的是`rs_EMP_DIFF` 5欄位新格式。
- 到職/復職/調入日「系統日 vs 生效日」比較邏輯（第113-125行），已用需求書「馬映佳」範例驗證，**現有邏輯已正確符合規格，不需修改**。

## ⚠️ 已確認的實際資料庫現況（使用者提供查詢結果）

### `sethq.job_employeediff_tmp` 實際schema

確認：實際DB表結構與現有C# `JobEmployeeDiffModel` 完全一致，**目前尚未有** `change_type`/`effective_date`/`extra_value` 欄位。→ 確認第1節的DB migration（新增這3個欄位）是必要的，且不會撞名。

### `job_config_setting_dev1` 中 `Job_UpdateEmployeeDiff` 現況 ⚠️重大發現

實際設定：`Table=Job_EmployeeDiff_tmp`、**`FTPFileName`/`ManualFileName` = `EMP_DIFF{0}.TXT`（⚠️無`rs_`前綴）**、`HeaderCount=8`、`IsChkFile=Y`、`IsDecode=Y`、`Split=|`，且 `Col1~Col16` 完整對應SAP介面規格書標註「不需此檔」的舊版16欄位「員工異動主檔(EMP_DIFF)」格式（`diff_status`, `empname`, `costid`, `empid`, `ptft`, `sapaname`, `effectivedate`, `expiredate`, `sex`, `sapteamid`, `identification`, `identification_type`, `sap_transfer_code`, `transfer_in_date`, `transfer_out_date`, `realonboarddate`）。

**這代表**：目前正式環境的 `Job_UpdateEmployeeDiff` 實際上還設定成抓取/解析舊版棄用格式的檔案，尚未切換到RA006要求的新版`rs_EMP_DIFF`5欄位格式。這不只是「Model/Dao新舊並存」的程式碼問題，**`job_config_setting_dev1`本身的設定資料也需要新增一組對應新格式的設定**（檔名改為`rs_EMP_DIFF{0}.TXT`，欄位改為5欄），且切換時機需與SAP端實際何時停發舊檔/改發新檔同步。

### `system_config_m` 相關設定現況

- `UpdateEmployee.TeamTypeWhiteList` / `UpdateOrganization.TeamTypeWhiteList` / `UpdateGPS.TeamTypeWhiteList` 皆為 `1,2,3,9`，**三處都需要新增`0`**（數位大樓），否則三個Job都會在白名單過濾階段排除該分類員工。
- `UpdateEmployee.IsMailPWD = N`（2026/5/20起）— **確認帳號啟用密碼通知信已停用**，符合規格書要求，此項不需再處理。

- `CheckAbnormalData.WhiteBlackList` 列出多個 `UpdateMasterData.Job_*` 全名 — 這是 **`NotifyMonitorJob`專案的Job執行監控通知機制**（見下方說明），與RA006要求的「業務資料異動提醒信」性質不同，是另一條軌道。

## ⚠️ 已確認的實際寄信程式碼架構（`NotifyMonitorJob`專案，使用者指定僅供參考）

使用者提供了 `C:\Users\LiuDe-Hong\Documents\TESTTTT\NotifyMonitorJob` 專案，明確要求：**規格書所有需要寄信的功能都要參考這套程式碼的寫法**。詳細讀取後確認，這與 `UpdateMasterData` 現有的 `mail_normal_m` + `MailTemplateService` 機制是**完全不同的兩套架構**：

| xlsx附件產生 | 無（尚未實作） | **NPOI(v2.6.2)** 的`XSSFWorkbook`，透過`ItemReflect<T>`反射模式動態把`List<T>`轉Excel欄位（`Service/CheckNotifyUserAbnormalService.GetNotifyUserAbnormalMailExcel()`），含中文欄寬自動調整 |
| 附件壓縮加密 | 無 | `Tool/ZipTool.cs` 用 `ICSharpCode.SharpZipLib.Zip` 的`ZipOutputStream`+密碼保護（`Job_CheckNotifyUserAbnormal.cs`範例：xlsx byte[] → 加密zip byte[] → 當附件寄出） |
| HTML內文組立 | 反射佔位符`@{PropName}`樣板替換(`MailTemplateService`) | `Tool/HtmlTool.cs`的`GetModelToHTMLTable`(反射把`List<T>`轉HTML表格)+`GetTableCSS()` |
| 寄信紀錄 | 寫入`mail_normal_m`（寄送前的佇列） | 寄送成功後才寫入`sethq.mail_normal_log`（純稽核LOG，非佇列） |
| 重試機制 | 無（單次寫入佇列即結束） | `while(true)` + `Thread.Sleep(1000)` + 環境變數`PIC_JOB_RETRY`控制次數 |

## 依規格書可以完成的修改項目

### 1. `Job_EmployeeDiff_tmp` 資料表 + Model + Dao — 支援新5欄位格式（新舊並存）

- **DB migration**（需另外執行，非程式碼本身）：在既有 `sethq.job_employeediff_tmp` 新增可為NULL的欄位 `change_type`(varchar2)、`effective_date`(date)、`extra_value`(varchar)，舊欄位保留不動（已確認實際schema無撞名疑慮）。
- **`job_config_setting_dev1` 設定變更**：需要新增一組對應新格式的設定（`FTPFileName`/`ManualFileName`改為`rs_EMP_DIFF{0}.TXT`，`HeaderCount`依新表頭長度重新計算，欄位改為5欄：`Col1.Name=empname,Len=20`、`Col2.Name=empid,Len=8`、`Col3.Name=change_type,Len=2`、`Col4.Name=effective_date,Len=8,Type=date,Rule=yyyyMMdd`、`Col5.Name=extra_value,Len=1`）。**上線切換時機需與SAP端實際何時停發舊檔/改發新檔同步**。
- **`Model/EmployeeDiff/JobEmployeeDiffModel.cs`**：新增 `ChangeType`/`EffectiveDate`/`ExtraValue` 三屬性，舊屬性保留。
- **`Dao/JobEmployeeDiffDao.cs`**：新增 `GetJobEmployeeDiffMap()`，以 `(Empid, ChangeType)` 複合鍵查詢，用 `ROW_NUMBER() OVER (PARTITION BY empid, change_type ORDER BY id DESC)` 取每組最新一筆（符合規格書「後蓋前」規則），不動既有舊方法。

### 2. `EmployeeViceService.UpdateEmployee()` 核心比對邏輯 — 完成第100-111行的空白邏輯

在既有 `editEmpList`/`addEmpList`/`deleteEmpList` 三段 LINQ 架構中插入情境判斷：

- **PT/FT轉換**（`editEmpList`內，目前技術債用途的`Birthday`欄位）：`tmp_Emp.Ptft != EV.Ptft`時，查 `diffMap[(empid,"01")]`或`"02"`（依轉換方向）的`EffectiveDate`；查無資料則fallback用`JobDate`並記警示。
- **分類異動**（`editEmpList`內，`GroupType`有差異時）：查`diffMap[(empid,"11")]`的`EffectiveDate`作為身分轉換日（建議沿用`TransferUpdateTime`欄位存放，因目前模型沒有專屬欄位——**這是一個假設，需與需求方確認是否有更合適的欄位**）；`ExtraValue`（新分類代碼）與主檔`GroupType`交叉驗證，不一致時記警示。
- **主檔不存在→離職/留停/調出**（`deleteEmpList`）：依序查`diffMap[(empid,"09")]`（離職）→`"06"`（調出）→`"10"`（留停），取第一個命中當作`Expiredate`/`TransferUpdateTime`；三者皆查無資料時維持既有`jobEmployeeviceTDao.GetTransferUpdateTime(...)`的fallback（不中斷流程，只記警示）。優先順序`09>06>10`為假設，需求方未明確定義。
- **到職/復職/調入日**：不修改，已驗證符合規格。

### 3. 警示記錄機制（★依NotifyMonitorJob模式修訂）

- `EmployeeViceService.UpdateEmployee()` 內收集「有差異但異動主檔查無對應[異動類型]資料」或「附加值與主檔不一致」的員編/情境清單（記憶體List，同先前設計）。
- **寄信方式改為參考`NotifyMonitorJob.Service.SMTPClientService`的直接寄信模式**：在`UpdateMasterData`專案內新增類似的`SMTPClientService`（或直接共用/複製該類別），用`MailKit`組`MimeMessage`直接同步寄出HTML內容的警示信，設定改走環境變數（比照`PIC_JOB_SMTPServer`/`_Port`/`PIC_JOB_MAILFROM`等命名慣例，新增`PIC_JOB_DIFFWARNING_MAILTO`等專屬環境變數）。
- 內文用類似`HtmlTool.GetModelToHTMLTable`的反射方式把警示清單轉成HTML表格。
- **不使用**`mail_normal_m`/`MailTemplateService`（這是`UpdateMasterData`原有機制，僅帳號啟用信在用，本次新功能依使用者指示改走`NotifyMonitorJob`模式）。
- **需要`UpdateMasterData.csproj`新增`MailKit`套件相依性**（版本比照`NotifyMonitorJob.csproj`的`4.7.1.1`）。

### 4. 分類0（數位大樓）設定檢查清單

- **已確認**：`system_config_m`的`UpdateEmployee.TeamTypeWhiteList`、`UpdateOrganization.TeamTypeWhiteList`、`UpdateGPS.TeamTypeWhiteList` 目前皆為`1,2,3,9`，**三處都需要新增`0`**，否則數位大樓員工在白名單過濾階段就會被三個Job（員工/組織/GPS主檔）分別排除轉入。這是三筆資料異動（非程式碼），建議列入上線前部署檢查清單。
- `EmployeeViceService.cs`「後勤成本中心轉換」判斷式`(GroupType == "3" || GroupType == "9")`是否要納入`"0"`，待需求方確認數位大樓是否算後勤類，暫不擅自變更。
