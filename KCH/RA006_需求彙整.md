## 1. 需求背景與目標

因應 SAP 端調整員工主檔規格，後勤考勤系統的「身分轉換」判斷功能需要調整。系統會收到：

1. **員工每日在職主檔（全件）**：SAP 每日提供，範圍為當天在職員工全量資料。分類新增 `0`：數位大樓。
2. **員工異動主檔（差異）**：SAP 每日提供前一天有異動的資料，異動類型含 `01/02/06/09/10/11`。

---

## 2. 系統身分轉換情境判斷規則（RA006 核心）

### 2.1 兩份主檔基本規則

- **員工每日在職主檔**：SAP 必須每日提供，範圍是當天在職員工，分類新增 `0`：數位大樓。
- **員工異動主檔**：SAP 必須每日提供，範圍是前一天有異動的資料，異動類型 `01`PT→FT(含建教生) `02`FT(含建教生)→PT `06`調出超商 `09`離職 `10`留停 `11`分類異動，皆需提供。
  - 新增分類（同在職主檔，新增 `0`：數位大樓），新增異動類型 `11`：分類異動。
  - 2026/4/21 會議說明：不增加「SAP建檔日」欄位。
  - 若多筆[異動類型]與[異動生效日]相同，系統以**後蓋前**方式取代。
  - 若發生缺檔，需由發生問題那一天開始逐日轉入至當天為止；前一天若無異動記錄，亦必須拋檔（表頭0筆、表身空白）。

### 2.2 本次改版重點（★核心邏輯）

1. 不同情境皆由**[員工每日在職主檔]與系統比對出差異**，才會從**[員工異動主檔]**找相關資料：
   - 若[在職主檔]未與系統比對出差異，則**忽略**[異動主檔]資料。
   - 若[異動主檔]資料中未找到相關日期，則會影響歷程，並於**提醒信件中補充**資料呈現「未有相關日期」。
2. 本次改版**僅影響**員工各情境變化之**日期判斷基準**，其餘判斷皆依原系統邏輯進行，不作調整。

### 2.3 比對方式（主檔情況 → 應查的異動類型）

| 主檔情況 | 應找的[異動類型]的[異動生效日] |
|---|---|
| PT→FT | `01`：PT→FT(含建教生) |
| FT→PT | `02`：FT(含建教生)→PT |
| 主檔不存在 | `06`調出超商 / `09`離職 / `10`留停 |
| 分類變更 | `11`：分類異動 |

### 2.4 各情境系統判斷/寫入規則彙整

| 情境 | 系統判斷規則 | 系統寫入規則 | 排班 | 申請單 |
|---|---|---|---|---|
| **新進人員到職** | 在職主檔出現新進員工 | 到職日 = [到職/復職/調入日]（例：人事令4/1生效，SAP於4/3才提供主檔，到職日仍=4/1） | 僅到職日(含)後可排班 | 僅可申請到職日(含)後之申請單 |
| **離職** | 在職主檔沒有該員工資料 | 離職日 = 異動主檔[09]的異動生效日 | 離職日(含)後保留排班(避免SAP錯誤無法隔日復原) | 離職日(含)後申請單可保留 |
| **離職後復職** | 在職主檔於復職日前或當日提供資料 | 到職日：系統日<[到職/復職/調入日]時維持舊到職日；系統日>=時才更新為新到職日（例：馬映佳4/7復職，SAP提早於4/2給資料，4/2-4/6到職日仍為舊值2025/1/13，到4/7當天才更新為4/7） | 僅到職日(含)後可排班 | 僅可申請到職日(含)後之申請單 |
| **留職停薪** | 在職主檔沒有該員工資料 | 留停日 = 異動主檔[10]的異動生效日（寫入離職日欄位） | 留停生效日(含)後保留排班 | 留停生效日(含)後申請單保留 |
| **留停後復職** | 在職主檔出現員工資料 | 到職日邏輯同「離職後復職」 | 僅復職日(含)後可排班 | 僅可申請復職日(含)後之申請單 |
| **PT/FT轉換** | 在職主檔切換身分(PT/FT) | 身分轉換日 = 異動主檔[01或02]的異動生效日 | 依分類排班邏輯（見2.6節） | 不受影響，可申請過去與未來單據 |
| **調任BU(調出)** | 在職主檔沒有該員工資料 | 調出日 = 異動主檔[06]的異動生效日（寫入離職日） | 調出生效日(含)後保留排班 | 調出生效日(含)後申請單保留 |
| **回任公司(調入)** | 在職主檔出現員工資料 | 調入日邏輯同「離職後復職」（到職日） | 僅調入日(含)後可排班 | 僅可申請調入日(含)後之申請單 |
| **分類異動** | 在職主檔變更為新分類 + 異動主檔判斷分類變更日 | 身分轉換日 = 異動主檔[11]的異動生效日 | 依轉換身分角色及區間套用對應排班規則（見2.6節） | 不受影響，可申請過去與未來單據 |

### 2.5 全域補充規則（適用所有情境）

- **考勤異常顯示判斷**：僅當員工日期落在「到職日(含)之後，離職日(不含)之前」且在系統「考勤異常啟用日」之後，才進行判斷與顯示。離職僅影響異常判斷結束日，不影響主管/人資查詢過往資料或簽核。
- **系統發信規則**：
  - 帳號啟用通知信：僅到職日隔日(+1日)起、且有效Email存在時發送。**已安排於5月版上線後停止發送**。
  - 一般考勤異常通知信：日期落在到職日(含)~離職日(不含)之間 + 考勤異常啟用日之後才發送。留停期間比照離職不發信；留停後復職比照新人到職處理。
  - 特休提醒通知信：僅「有效在職期間」發送。
- **系統登入權限**：僅到職日(含)~離職日(不含)期間開放；不符合時 Oauth/帳密登入分別有對應錯誤訊息（2026/5/28會議調整文字內容，含分機2981/2982）。
- **報表/月結規則**：PT工時報表、PT工時月結檔僅統計PT身分期間工時，依到職日(含)~離職日(不含)期間所記錄的不同身分區段分段呈現。

### 2.6 分類轉換代碼與排班行為（R.其他）

**分類代碼**：`0`數位大樓 `1`商場 `2`聯服 `3`地區 `4`門市 `9`其他

**新身份生效日及之後的系統排班行為**：

| 轉入身份 | 系統行為 |
|---|---|
| 數位大樓(0) | 刪除非後勤排班，排入特別班(數位)，移除成本中心判斷條件，改用分類0判斷 |
| 商場(1) | 保留排班，由主管改動班表 |
| 聯服(2) | 保留排班，由主管改動班表 |
| 地區(3) | 刪除非後勤排班，排入特別班 |
| 門市(4) | - |
| 其他(9) | 刪除非後勤排班，排入特別班 |

**補充（4/27信件）**：
- 聯服特殊人員（郭O廷、郭O婷）因與超商約定班表，需由超商通知PIC後人工處理。
- 區顧問/區經理判斷邏輯改用 JOBCODE 判斷（區顧問=`00000017`，區經理=`00000013`），移除職稱字元判斷。當天班表為「特別班」且該員為區顧問/區經理時，提供額外班別可調班：`13:00~22:00`、`17:00~02:00`、`23:00~08:00`、特別班。

### 2.7 主檔轉入檢核邏輯與警示機制（5/19會議新增）

| 主檔資料 | 基本檢核（失敗時中止整份轉入） | 新增需寄警示並排除轉入邏輯（維運項目累加機制） |
|---|---|---|
| 員工在職主檔 | 1.檔名日期為當天 2.表頭符合(識別碼/日期/資料總數/長度) 3.表身欄位數量一致 4.資料符合欄位型態/長度限制(如到職日須為日期) 5.員工編號唯一值 | 1.職稱為空 2.單位代號不存在於系統內 |
| 員工異動主檔 | 同上1-4項 | 1.由在職主檔觸發時才會從異動主檔抓資料，不存在時寄警示（例：某員PT轉FT但異動主檔沒有轉換日資料） |

- 2026/5/28會議補充：警示信主旨由超商提供，每天09:00寄出，問題明細於附件提供。
- 送審中申請單仍用系統最新匯報關係決定簽核者，不因離職而不寄簽核信（除非Email失效）。
- 總經理為第一層主管者應設自動簽核；未設定時仍依匯報關係由總經理簽核（但不發信給總經理）。

### 2.8 人資通知信件發送規則（人員異動提醒名單）

- 主旨：【人員異動提醒名單】
- 附件檔名：`人員異動提醒名單yyyyMMdd.xlsx`，欄位：層級單位、單位、員編、姓名、情境、補充資料
- 內容：本日提醒名單總數：{數字}
- 邏輯：先用[在職主檔]與系統比對，若不同再用[異動主檔]找出對應[異動類型]的[異動生效日]後，異動主檔有轉入系統都寄出提醒名單。
- 情境涵蓋：新進人員到職、離職、離職後復職、留職停薪、留停後復職、PT/FT轉換(FT→PT / PT→FT)、調任BU(調出)、回任公司(調入)、分類異動、職稱異動、單位異動、姓名異動、Email異動，各情境有對應補充資料格式（原/新日期或原/新分類區間，皆為 yyyy/MM/dd）。

### 2.9 額度(Quota)相關

- 所有額度與人員異動無關，除非人工介入，僅以SAP最新版本作為期初額度計算。
- 4月版：每月1日全件轉入時，會刪除不存在於全件中但系統仍生效的SAP_UK（5/1日刪除額度已復原）。
- 未來改為：新增一筆當月版本但總額度為0的SAP_UK資料，取代直接刪除。

### 3.2 ⚠️ 重大風險提醒：兩種「異動」檔案不可混淆

規格書內有兩種名稱相似但完全不同的異動檔案，這是本次比對中最容易出錯的地方：

| | 員工異動記錄（實際使用） | 員工異動主檔（已棄用） |
|---|---|---|
| 檔名前綴 | `rs_EMP_DIFF...TXT`（有`rs_`前綴） | `EMP_DIFF...TXT`（**無**`rs_`前綴） |
| 欄位數 | **5欄**：姓名/員編/異動類型/異動生效日/分類 | **16欄**：狀態(I/U/D)/姓名/所屬門市/員編/人員類別/職稱/到職日/離職日/性別/組織單位代碼/ID/ID類別/到職復職調入日/異動日/職稱代號/Email |
| 目錄頁標註 | 每天提供，SAP→後勤考勤 | **「不需此檔」** |
| 異動類型欄位 | 01/02/06/09/10/11（2碼） | 無此概念，用狀態I/U/D表示 |

**重要發現**：現有 `UpdateMasterData` 專案內的 `JobEmployeeDiffModel`（欄位含 `DiffStatus`(I/U/D)、`SapTransferCode`(值域B8/C1/B5)、`TransferInDate`/`TransferOutDate`）明顯對應到規格書標註「不需此檔」的**舊版棄用格式**，而非目前SAP實際提供、也是本次RA006規格重點的 `rs_EMP_DIFF` 5欄位格式。任何後續程式碼修改都必須先確認是否用錯了對應格式。
